    Guide de Développement

    Vue d'ensemble

    Philosophie

    Notre workflow est conçu pour être : - Simple à comprendre - Facile à suivre - Efficace en équipe - Maintenable dans le temps

    Outils nécessaires

    • Git
    • VS Code (recommandé)
    • CMake
    • Compilateur C++

    Cycle de développement

    1. Préparation

    1. Choisir une issue ou en créer une
    2. Discuter de l'approche en commentaires
    3. Assigner l'issue à soi-même

    2. Développement

    # Créer une branche
git checkout -b feature/ma-fonctionnalite

# Développer avec tests
# Commit régulièrement
git commit -m "feat(scope): description"

    3. Validation

    • Tests unitaires passent
    • Style de code respecté
    • Documentation à jour

    4. Pull Request

    • Description claire
    • Références l'issue
    • Screenshots si UI
    • Tests mentionnés

    Standards de Code

    Commits

    ✅ BIEN : 

    feat(grid): add toroidal mode
fix(ui): correct cell color update
docs(readme): update installation guide

    ❌ À ÉVITER : 

    update code
fix bug
wip

    Branches

    • main : production
    • develop : développement
    • feature/* : nouvelles fonctionnalités
    • fix/* : corrections de bugs

    Revue de Code

    Checklist du reviewer

    • [ ] Code lisible et commenté
    • [ ] Tests présents et passent
    • [ ] Pas de régressions
    • [ ] Documentation à jour
    • [ ] Style respecté

    Feedback constructif

    ✅ BIEN : 

    Suggestion : utiliser std::vector ici pour la gestion mémoire

    ❌ À ÉVITER : 

    Ce code est mauvais

    Intégration Continue

    Actions automatiques

    name: CI
on: [push, pull_request]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Build & Test
        run: |
          mkdir build
          cd build
          cmake ..
          make
          ./tests/unit_tests

    Vérifications

    • Build ✅
    • Tests ✅
    • Style ✅
    • Documentation ✅

    Versions et Releases

    Versioning

    • MAJOR : changements incompatibles
    • MINOR : nouvelles fonctionnalités
    • PATCH : corrections de bugs

    Publication

    1. Mettre à jour CHANGELOG.md
    2. Créer un tag Git
    3. Build sur toutes plateformes
    4. Publication sur GitHub
